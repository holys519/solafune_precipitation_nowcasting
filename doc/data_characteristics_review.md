exp001〜exp011 の結果 (`doc/public_scores.md`, `outputs/analysis/*/analysis_summary.json`)、
`outputs/g_eda/exp001/` のEDA成果物、および本レビューのために新たに実行した

## 0. ルール運用前提 (2026-07-09)

本レビューの実験方針では、`doc/competition_rules.md` の Working Assumption に従い、
Solafuneから配布されたtrain/evaluation CSV、衛星GeoTIFF、train target、sample/evaluation
template、およびそれらから作成した特徴量・モデル入力・後処理特徴は原則利用可とみなす。

したがって、同一locationの隣接evaluation行、successor/predecessor行の衛星観測、
モデル予測同士の時間平滑化、配布データ内で完結する画像特徴マッチングは、外部データを
使わない限り実験対象に含める。明らかな外部データセット、外部気象データ、外部地理座標、
利用不可ライセンスの重みは使わない。

運営から追加の明示的な禁止・確認回答が出た場合は、その回答を優先して方針を更新する。

## 1. 新たに確認したデータ特性

### 1.1 各locationは「約1ヶ月の連続エピソード」である【最重要】

**思考過程**: exp009 のper-location OOFで dhaka/bihar (モンスーン地帯) の
target_mean がほぼ0だった。気候学的にありえないので「乾季だけサンプリングされて
いるのでは」と疑い、まず3地点の月分布を、次に全地点の datetime 範囲を調べた。

**結果** (train/eval CSVの datetime を地点別に集計):

- train 20地点中 15地点が **単一月** のみ (例: bihar/dhaka/guangdong/jakarta =
  2023-01のみ、cape_town/gaza_province = 2026-01のみ)。残りも2ヶ月 (france のみ5ヶ月)。
- eval 18地点も同様: 16地点が単一月。
- 各地点約1,440〜1,488行 = **30分刻みの連続観測1ヶ月分**。つまり1地点 = 1本の
  「30分刻み降水ビデオ」。

**含意**:

1. 40,686行という行数は見かけで、**独立な気象レジームは実質 ~20個** (train)。
   同一エピソード内の行は同じ嵐を繰り返し見ているだけで強く自己相関している。
2. fold間の気候差が激烈 (exp009 の fold別 target_mean: 0.104〜0.445、
   fold RMSE: 0.656〜1.476)。**CV std 0.26 は「モデルが不安定」ではなく
   「foldの雨量気候が違う」が正体**。現行の5-fold GroupKFold のCV mean/std は
   実験間比較のS/Nが低い。
3. exp009 の successor-row が効いた理由もこれで説明できる: successor行は
   文字通り「同じビデオの次のフレーム」なので、ほぼ同一シーンの追加観測になる。

### 1.2 train/evalエピソードは同一時間窓・近傍地域でペアになっている

**思考過程**: 地点別 datetime 範囲を並べたとき、eval の north_sumatra が train の
aceh と **開始日・終了日・行数まで完全一致** (2023-01 + 2025-11, 2926行) している
ことに気づいた。`outputs/g_eda/exp001/train_eval_feature_shift.csv` でも
north_sumatra の最近傍train地点は aceh (特徴距離0.014) であり、時間・空間の両方で
ペア構造がある。他にも sylhet↔dhaka/bihar (2023-01)、mekong_delta/quang_nam↔
jakarta/guangdong (2023-01)、northeast_malaysia/sri_lanka↔hat_yai (2025-11)、
tanganyika/niger_state↔kinshasa (2025-04) など。

そこで「同時刻の隣接train GPMターゲットがeval予測の手がかりになる (配布データ内で
完結)」可能性を検証するため、GeoTIFFの座標タグを読んだ。

**結果**: GPM・衛星ファイルとも GeoKey はなく ModelTransformationTag (34264) のみ、
かつ **中身は恒等行列** (`[1,0,0,0, 0,1,0,0, ...]`)。**座標は運営により匿名化
されている**。メタデータから空間隣接は特定できない。

**含意**:

- 同時刻・近傍という構造自体は確実に存在する (時間窓一致 + g_eda特徴距離)。
  画像内容ベースのマッチング (同時刻の衛星画像同士の相互相関でタイル位置関係を
  推定) は原理上可能で配布データ由来なので実験対象に含める。ただしエンジニアリング
  コストが高くリターン不確実なため、研究枠 (§4.7) としてのみ推す。
- 匿名化されている以上、位置座標そのものを特徴に使う道はない。ただし
  **タイル内の緯度は不明でも「日変化の位相」は画像から推定可能** (§1.4)。

### 1.3 メトリクスの誤差予算: mid-rain + heavy-rain タイルが87%を占める

**思考過程**: exp008 のdrizzle除去 (光量閾値0.25) の改善が小さかった理由と、
次にどこを攻めるべきかを定量化するため、exp009 OOF の per-sample `tile_rmse` を
タイル雨量レジーム別に分解した (tile_rmse はサンプル平均なので寄与が線形に分解
できる — これがこの分解を選んだ理由)。

**結果** (exp009 OOF, 40,686タイル):

| タイルのtarget_mean | タイル数 | 平均tile_rmse | **メトリクス寄与** |
| --- | ---: | ---: | ---: |
| 0 (完全無雨) | 13.3% | 0.024 | **0.5%** |
| <0.1 (微雨) | 45.4% | 0.176 | **12.8%** |
| 0.1–1 (中雨) | 34.1% | 0.968 | **53.0%** |
| ≥1 (強雨) | 7.2% | 2.934 | **33.7%** |

**含意**: スコアの87%は中雨〜強雨タイルで決まる。無雨タイルはほぼゼロ寄与。
つまり **rain/no-rain検出の改善余地は構造的に小さく (CSI plateau 0.44-0.46 を
押してもリターン限定)、雨量回帰の精度、特に中〜強雨の振幅が本丸**。exp008の
drizzle後処理の効果が小さかったのは必然 (触れる誤差が13%しかない)。

### 1.4 昼夜で誤差が系統的に変わる (3衛星すべて)

**思考過程**: 16バンドのうち可視・近赤外バンドは夜間に情報を持たない、という
センサー物理から「夜間の誤差が大きいはず」と仮説を立て、UTC時刻別の平均
tile_rmse を衛星別に集計した。衛星ごとに現地時間帯が違うので、極小の位置が
各衛星の現地昼間に対応するかで判定できる。

**結果**: 誤差極小は himawari 05-06 UTC (現地~14時)、goes 14-16 UTC (現地~9-11時)、
meteosat 08-11 UTC (現地昼) — **3衛星とも現地昼間に誤差最小、夜間に最大**
(himawari で夜間 +35%程度)。

**含意**: モデルは昼夜という潜在モードを入力から明示的には知らされていない
(可視バンドが暗いことから間接的に推測するしかない)。外部座標は使わず、
**UTC hour × satellite_target、または可視band輝度から作る昼夜proxyを入力チャネルに
追加する** のは、配布CSV/画像から導出できるので安価。ただし対流の日周期
(午後の雷雨) との交絡があるので、
「同じ雨量強度で昼夜の誤差を比較する」条件付き分析 (§5 A-2) で切り分けてから
特徴設計するのが筋。

### 1.5 強雨の振幅を系統的に大きく過小評価している

**思考過程**: §1.3 で強雨タイルが誤差の1/3を占めると判ったので、その誤差の
性質を見るため pred_max / target_max をtarget_max ビン別に集計した。

**結果**: target_max ∈ [10,100) のタイルで **pred_max/target_max = 0.25**
(ビンが下がるほど改善: [0.5,2)で0.45)。強い雨ほど振幅をつぶしている。

**含意**: MSE最適解は条件付き平均なので極値の過小評価自体は想定内だが、0.25は
「平均への回帰」だけでなく**空間的な平滑化 (ピークのスミア)** も疑われる。
どちらが支配的かで対策が変わる (前者→ log1p変換・強度重み付き損失、後者→
解像度・デコーダ設計・sharpness正則化)。切り分け分析を §5 A-3 に定義した。

### 1.6 RMSEは地点の雨量気候にほぼ比例する — スキルと気候の分離が必要

**結果** (exp009 per-location OOF): RMSE上位は central_vietnam 2.06 (target_mean
0.52)、下位は dhaka 0.04 (同 0.000) と、**誤差がほぼ気候量に比例**。per-location
bias も -0.29 (central_vietnam) 〜 +0.24 (ecuador) と大きくばらつく。

**含意**: (1) 生RMSEでの実験間比較はfold構成の気候に振り回される。地点別
`RMSE / zero-baseline RMSE` のようなスキル比を registry に併記すべき
(`doc/experiment_tracking_design.md` に列追加)。(2) per-location bias は未知の
eval地点では直接補正できない。補正するなら「地点」でなく「レジーム」(衛星・
雨量帯・昼夜) を条件にする。

## 3. CV設計への提言

現行: 20エピソードを5-foldに分割 (1foldあたり4エピソード)。

問題: (1) fold気候差でCV分散が巨大 (§1.1)、(2) epoch選択も fold内validで行うため
選択ノイズが大きい、(3) OOF全体値がfold構成に依存し、exp010のようなOOF/public
乖離を検出しにくい。

提言 (コスト順):

1. **即効・ゼロコスト**: registry に per-location スキル比 (`RMSE/zero-baseline`)
   と per-satellite 内訳を必ず記録し、実験比較は「同一fold同士の対比較」+
   「スキル比」で行う。生CV meanの差での判断をやめる。
2. **中コスト**: leave-one-location-out CV (20 fold)。モデルが小さい
   (CompactUNet系) ので計算上は現実的。全地点のOOFが揃い、地点レベルの
   対応のあるt検定的な比較ができる。まず「現行チャンピオン vs 挑戦者」の
   決定戦にだけ使う運用でもよい。
3. **fold再設計**: 気候量 (地点別target_mean) で層化したGroupKFold。fold間の
   雨量バランスを揃え、既存の5-fold運用のままS/Nを改善する。

## 4. 推奨アプローチ (優先度つき)

1. **G-016 (exp012: exp009×exp011統合)** — 既定路線を維持。両親とも単独で
   アンカー超え、直交する軸。
2. **中〜強雨の振幅回復** (新規・高優先): §1.3+1.5 より誤差予算87%がここ。
   まず A-3 (§5) でスミアか平均回帰かを切り分け、log1p ターゲット変換または
   強度重み付き損失 (research_survey §4 のintensity-aware loss) をexp012の上で
   1-fold A/B。two-head の amount 側だけに適用するのが安全。
3. **UTC hour / day-night proxy特徴** (新規): §1.4。外部座標を使わず、CSVのdatetime、
   satellite_target、または可視band輝度から作る。A-2 の切り分け後に投入。
4. **CV再設計 + スキル比メトリクス** (§3): モデル改善ではないが、以後の全実験の
   判断品質を上げる複利投資。registry設計 (I-001) と同時に入れるのが効率的。
5. **予測の時間平滑化** (exp008に実装済み・デフォルトoff): evalも30分刻み連続
   ビデオ (§1.1) なので、隣接予測との平滑化は理論上有効。ただし GPM 自体の
   30分自己相関 (A-1) を測ってから閾値を決める。
6. **G-017 parallax registration / G-018 文脈拡大** — 既存チケット通り。
   G-018は§1.1の構造から predecessor 側も同様に効くはずで、期待値は高い。
7. **研究枠: 画像内容ベースの train-eval 隣接探索** (§1.2): 当たれば同時刻の
   実GPMターゲットという最強の特徴が手に入るが、匿名化されており工数リスク大。
   1ペア (aceh↔north_sumatra) だけで試す小さなスパイクに留める。

## 6. 思考過程の総括 (メタ)

このレビューで使った推論の流れを明示しておく (再利用可能な手順として):

1. **矛盾から入る**: 「モンスーン地帯なのに無雨」「CV stdがmeanの25%」という
   気候学的・統計的な違和感を出発点にした。異常値は大抵、データ生成過程の
   未文書化な構造 (今回はエピソードサンプリング) を指している。
2. **誤差を予算として分解する**: どの施策も「触れる誤差の総量」が上限になる
   (Amdahl則と同じ発想)。tile_rmse が線形分解可能であることを利用して
   レジーム別寄与を先に出し、施策の期待値を序列化した。
3. **物理からの事前仮説で探索を絞る**: 可視バンドは夜ゼロ、静止衛星の
   パララックス、対流の日周期 — 物理的にありうる効果を先に列挙してから
   データで確認する方が、無方向の相関探しより速い。
4. **結果の後講釈をデータ特性と突き合わせる**: exp005の失敗はg_edaのmotion
   計測から事前に予測できた。「実験前にデータ特性から結果を予言してみる」
   ことを以後の実験計画の標準ステップにすべき。
5. **配布データ内で完結するかを即確認する**: 時間窓ペアを見つけた時点でGeoTIFF座標を
   読み、外部座標に依存しないことを確認した。画像内容ベースの隣接探索は可能性として
   残すが、工数リスクが高いので研究スパイクに留める。
